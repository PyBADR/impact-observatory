Overview Architecture

This Knowledge Graph unifies three operational domains — Insurance, Technical/Observability, and Intelligence/Events — into a single semantic layer built on top of the existing GCC Infrastructure Graph (42 nodes, 9 sectors) and Propagation Engine. It is designed to serve as the foundational data layer for AI reasoning, decision synthesis, and regulatory compliance tracing across the entire Impact Observatory stack.

Design Principles

Codebase Alignment
Every entity type maps to an existing Pydantic model, TypeScript interface, or graph_brain enum. No orphan abstractions — the graph is the runtime data model.
Deterministic Lifecycle
Signal → Event → Anomaly → Decision → Outcome. Every intelligence path is traceable through typed relationships with confidence scores and audit hashes.
Cross-Domain Bridging
Insurance claims link to geopolitical events. Deployment failures link to affected services and the GCC nodes they serve. The graph enables reasoning across operational silos.
Audit-First
SHA-256 content hashes, IFRS 17 group classifications, PDPL-compliant data lineage. Every node and edge supports provenance tracking for GCC regulatory requirements.

Graph Schema Design Neo4j 5.x

Constraint Architecture

Every node label has a UNIQUE constraint on its primary identifier. This ensures referential integrity and enables O(1) lookups by ID across the entire graph.

// ── 26 unique constraints (one per entity type) ──
CREATE CONSTRAINT customer_id_unique IF NOT EXISTS
  FOR (n:Customer) REQUIRE n.customer_id IS UNIQUE;

CREATE CONSTRAINT policy_id_unique IF NOT EXISTS
  FOR (n:Policy) REQUIRE n.policy_id IS UNIQUE;

CREATE CONSTRAINT deployment_id_unique IF NOT EXISTS
  FOR (n:Deployment) REQUIRE n.deployment_id IS UNIQUE;

CREATE CONSTRAINT signal_id_unique IF NOT EXISTS
  FOR (n:Signal) REQUIRE n.signal_id IS UNIQUE;

CREATE CONSTRAINT gcc_node_id_unique IF NOT EXISTS
  FOR (n:GCCNode) REQUIRE n.node_id IS UNIQUE;

// ... 21 more constraints (see knowledge-graph-schema.cypher)

Index Strategy

Three index tiers: temporal (range queries on timestamps), classification (enum/status filters), and composite (hot query path optimization).

// ── Temporal (7 indexes) ──
CREATE INDEX idx_claim_filed_at FOR (n:Claim) ON (n.filed_at);
CREATE INDEX idx_signal_received_at FOR (n:Signal) ON (n.received_at);
CREATE INDEX idx_deployment_created_at FOR (n:Deployment) ON (n.created_at);

// ── Classification (7 indexes) ──
CREATE INDEX idx_claim_status FOR (n:Claim) ON (n.status);
CREATE INDEX idx_gcc_node_sector FOR (n:GCCNode) ON (n.sector);

// ── Composite (3 indexes) ──
CREATE INDEX idx_claim_sector_status FOR (n:Claim) ON (n.sector, n.status);
CREATE INDEX idx_signal_domain_severity FOR (n:Signal) ON (n.impact_domain, n.severity);

Example Cypher Queries 9 Queries

Q1: Deployment Failures → Affected Services → Business Impact

MATCH (d:Deployment {state: 'ERROR'})-[:DEPLOYS]->(s:Service)
OPTIONAL MATCH (s)-[:POWERS_ANALYTICS_FOR]->(g:GCCNode)
RETURN d.deployment_id, d.commit_sha, s.name, s.status,
       collect(g.label) AS affected_gcc_nodes
ORDER BY d.created_at DESC;

Q2: Claims → Customer Risk Profiles (Fraud Detection)

MATCH (c:Customer)-[:OWNS_POLICY]->(p:Policy)-[:GENERATED_CLAIM]->(cl:Claim)
MATCH (c)-[:HAS_RISK_PROFILE]->(r:RiskProfile)
WHERE cl.status = 'OPEN' AND cl.amount_claimed_usd > 1000000
RETURN c.name, c.risk_tier, r.overall_score, r.geopolitical_score,
       cl.claim_id, cl.amount_claimed_usd, cl.fraud_score
ORDER BY cl.amount_claimed_usd DESC;

Q3: Cross-System Anomaly Detection (7-Day Window)

MATCH (a:Anomaly)
WHERE a.severity > 0.6 AND a.detected_at > datetime() - duration('P7D')
OPTIONAL MATCH (a)<-[:CAUSED_ANOMALY]-(e:Event)-[:AFFECTS]->(g:GCCNode)
RETURN a.anomaly_type, a.severity, a.deviation_factor,
       e.event_type, collect(DISTINCT g.label) AS affected_nodes
ORDER BY a.severity DESC;

Q4: Technical Events Correlated with Business Impact

MATCH (d:Deployment)-[:DEPLOYS]->(s:Service)-[:POWERS_ANALYTICS_FOR]->(g:GCCNode)
MATCH (g)<-[:AFFECTS]-(e:Event)<-[:TRIGGERS]-(sig:Signal)
WHERE d.created_at > datetime() - duration('P30D')
RETURN d.deployment_id, d.state, s.name,
       g.label, e.event_type, sig.severity
ORDER BY sig.severity DESC;

Q5: Full Signal → Outcome Lifecycle Trace

MATCH path = (sig:Signal)-[:TRIGGERS]->(evt:Event)
              -[:CAUSED_ANOMALY]->(anom:Anomaly)
              -[:PROMPTED_DECISION]->(dec:Decision)
              -[:PRODUCED_OUTCOME]->(out:Outcome)
RETURN sig.signal_id, sig.severity AS signal_severity,
       evt.event_type, anom.anomaly_type,
       dec.action, dec.loss_avoided_usd,
       out.classification, out.net_value_usd,
       length(path) AS lifecycle_hops;

Q6: Multi-Hop Propagation Path Analysis

MATCH path = (entry:GCCNode)-[:PROPAGATES_TO*1..5]->(terminal:GCCNode)
WHERE entry.node_id = 'hormuz'
RETURN [n IN nodes(path) | n.label] AS propagation_chain,
       [r IN relationships(path) | r.weight] AS weights,
       reduce(w = 1.0, r IN relationships(path) | w * r.weight) AS cumulative_weight
ORDER BY cumulative_weight DESC;

Q7: Insurance Exposure by GCC Event

MATCH (evt:Event)-[:TRIGGERED_CLAIMS_IN]->(g:GCCNode {sector: 'insurance'})
MATCH (cl:Claim)-[:CAUSED_BY_EVENT]->(evt)
MATCH (cl)<-[:GENERATED_CLAIM]-(p:Policy)<-[:OWNS_POLICY]-(c:Customer)
RETURN evt.event_type,
       count(DISTINCT cl) AS claim_count,
       sum(cl.amount_claimed_usd) AS total_exposure_usd,
       collect(DISTINCT c.name) AS affected_customers;

Q8: Fraud Detection — High-Risk Claims with Event Correlation

MATCH (cl:Claim)
WHERE cl.fraud_score > 0.5
OPTIONAL MATCH (cl)-[:CAUSED_BY_EVENT]->(evt:Event)
OPTIONAL MATCH (cl)<-[:GENERATED_CLAIM]-(p:Policy)<-[:OWNS_POLICY]-(c:Customer)
               -[:HAS_RISK_PROFILE]->(r:RiskProfile)
RETURN cl.claim_id, cl.fraud_score, cl.amount_claimed_usd,
       c.name, c.risk_tier, r.overall_score,
       evt.event_type,
       CASE WHEN evt IS NULL THEN 'NO_CORRELATED_EVENT'
            ELSE 'EVENT_CORRELATED' END AS correlation_status;

Q9: Deployment Verification Audit Trail

MATCH (d:Deployment)-[:BUILT_BY]->(b:Build)-[:BUILT_FROM]->(c:Commit)
      -[:COMMITTED_TO]->(r:Repository)
WHERE d.target = 'production'
RETURN d.deployment_id, d.state, d.verified,
       d.verification_fingerprint,
       c.sha, c.message, c.committed_at,
       b.duration_seconds, r.full_name
ORDER BY d.created_at DESC LIMIT 10;

Use Cases 6 Core

1. Risk Analysis — Portfolio Exposure Under GCC Stress

Traverse from a Scenario through SimulationRun to impacted GCCNodes, then cross into the insurance domain to quantify aggregate portfolio exposure. The graph enables answering: "If Hormuz closes, what is our total claims exposure across all marine hull policies?"

This query path spans three domains: Propagation → GCC Infrastructure → Insurance. It replaces manual spreadsheet analysis that previously took 2–3 days with a sub-second graph traversal.

Enabled by: Q6 (Propagation Path) + Q7 (Insurance Exposure)

2. Fraud Detection — Event-Uncorrelated Claims

Claims filed during or after a GCC event (e.g., Hormuz disruption) that cannot be linked to that event via the CAUSED_BY_EVENT relationship are statistically more likely to be fraudulent. The graph identifies claims with high fraud_score that lack event correlation — a pattern invisible to traditional flat-table queries.

By combining risk profile scores (financial, operational, geopolitical) with claim-event correlation status, investigators can prioritize review of the highest-risk claims first.

Enabled by: Q2 (Claims → Risk) + Q8 (Fraud Detection)

3. System Observability — Deployment Impact Chain

When a deployment fails (state: ERROR), the graph traces the impact: Deployment → Service → Environment → GCCNode. If the frontend service goes down, which GCC sectors lose their analytics dashboard? If the backend API fails, which simulation runs are blocked?

This enables SRE teams to understand business impact of technical incidents in real time, not after a post-mortem.

Enabled by: Q1 (Deployment Failures) + Q9 (Audit Trail)

4. Impact Analysis — Cross-Sector Contagion Modeling

The PROPAGATES_TO relationship with weight, channel, and decay properties enables variable-depth graph traversal to model how a shock in one sector (e.g., maritime) cascades through banking, insurance, and fintech. The cumulative_weight reduction along the path quantifies contagion attenuation.

This directly implements the propagation model from config.py (PROP_BETA=0.65, PROP_LAMBDA=0.05, PROP_CUTOFF=0.005) as first-class graph structure rather than procedural code.

Enabled by: Q6 (Propagation Paths)

5. Customer Profiling — Multi-Dimensional Risk Intelligence

Each Customer connects to RiskProfile (multi-score assessment), Policy (exposure), Sector (business classification), and Country (geography). An AI agent can traverse from a customer through their policies to see which GCC events affected their coverage territory, what claims were filed, and how the risk profile should be updated.

This supports dynamic repricing: after a Hormuz event, all marine hull policies for UAE-based maritime companies should be flagged for premium review.

Enabled by: Q2 (Claims → Risk) + Q7 (Insurance Exposure)

6. Decision Support — Full Lifecycle Reasoning

The Signal → Event → Anomaly → Decision → Outcome chain is the graph's core intelligence lifecycle. Each Decision node records its reasoning_chain and contribution percentages (graph_contribution_pct, propagation_contribution_pct, rule_contribution_pct), enabling explainable AI that shows auditors exactly why a recommendation was made.

Outcome classification (TRUE_POSITIVE, FALSE_POSITIVE, etc.) creates a feedback loop: decisions that produced FALSE_POSITIVE outcomes indicate model drift, triggering recalibration of the decision brain's weighting.

Enabled by: Q5 (Lifecycle Trace)

Extensibility Plan Roadmap

Phase 1: External Macroeconomic Data

Add :MacroIndicator nodes (oil price, CPI, interest rates, FX rates) with :INFLUENCES relationships to :GCCNode and :Sector. Data sourced from signal_intel/ economic feed adapters. Enables queries like "How does a $20/barrel oil price drop propagate through the Saudi banking sector to insurance combined ratios?"

Schema extension: CREATE (:MacroIndicator {indicator_id, type, value, unit, source, observed_at}). No existing schema changes required — pure additive.

Phase 2: Additional Insurance Products

Add :InsuranceProduct node (already reserved in constraints) with product-specific sub-labels: :MotorProduct, :HealthProduct, :PropertyProduct, :LifeProduct. Each product links to :Policy via :ISSUED_UNDER and inherits :Coverage templates via :OFFERS_COVERAGE.

This enables product-level portfolio analytics: "What is our total motor insurance exposure across UAE during a traffic infrastructure failure scenario?"

Phase 3: More Event Sources

The :Signal node already supports signal_type (GEOPOLITICAL, ECONOMIC, INFRASTRUCTURE) and source (RSS_FEED, JSON_API). Adding new feed types (SATELLITE, IoT_SENSOR, SOCIAL_MEDIA, REGULATORY_FILING) requires only new enum values and signal_intel/ adapter plugins — the graph schema is unchanged.

For high-volume sources, add :SignalBatch aggregate nodes that summarize 100+ signals into a single node with statistical properties (mean_severity, max_severity, signal_count).

Phase 4: AI/LLM Agent Integration

Add :Agent nodes representing AI agents (Decision Brain, Fraud Detector, Pricing Engine) with :RECOMMENDED relationships to :Decision nodes. Each recommendation edge carries model_version, confidence, latency_ms, and prompt_hash for LLM audit trailing.

Add :Conversation and :ReasoningStep nodes for LangGraph chain-of-thought tracing, linked to the Decisions they informed. This enables post-hoc analysis of which reasoning paths produce the best Outcome classifications.

Schema: (:Agent)-[:RECOMMENDED {confidence, model_version}]->(:Decision) and (:ReasoningStep)-[:NEXT_STEP]->(:ReasoningStep)

Phase 5: Multi-Tenant SaaS Isolation

Add :Tenant nodes with :BELONGS_TO_TENANT relationships on all insurance entities (Customer, Policy, Claim). Enforce data isolation via Neo4j's role-based access control (RBAC) with per-tenant database users, or via property-level filtering (WHERE n.tenant_id = $tenantId) on all query paths.

For the ZhongAn/Lemonade multi-tenant model, each tenant gets a logical subgraph partition while sharing the GCC Infrastructure and Technical domains as read-only shared context.

Phase 6: Real-Time Streaming

Connect Neo4j Streams (Kafka connector) to the signal_intel/ ingestion pipeline. New :Signal nodes and :TRIGGERS relationships are created in real time as feeds are processed. The graph becomes a living representation of the GCC risk landscape, not a periodic snapshot.

Add :GraphVersion nodes for temporal versioning — enabling point-in-time queries: "What did the graph look like 24 hours before the Hormuz disruption?"

Scaling Considerations

Node Growth Projection
Year 1: ~5K nodes (42 GCC + insurance entities). Year 2: ~50K (multi-tenant customers + signals). Year 3: ~500K (real-time streaming signals + LLM reasoning chains). Neo4j Enterprise supports billions of nodes — no architectural ceiling.
Query Performance
All hot paths are covered by composite indexes. PROPAGATES_TO traversals are bounded to 5 hops (matching PROP_CUTOFF). APOC procedures available for Dijkstra/A* if weighted shortest path is needed at scale.
Backward Compatibility
Every extension phase is purely additive — new labels, new relationship types, new properties. No existing schema is modified. The IF NOT EXISTS clause on all constraints ensures idempotent re-runs.
